1. Données brutes → HDFS /data/raw/

NYC Taxi TLC — fichiers Parquet (~500 MB/mois). Stockage distribué sur le cluster Hadoop.

Les fichiers yellow_tripdata_2023-0X.parquet sont téléchargés depuis le CDN TLC et uploadés dans HDFS via hdfs dfs -put. HDFS découpe chaque fichier en blocs de 128 MB répliqués 3× sur les DataNodes.

⚙️ Commande clé : hdfs fsck /data/raw/ -files -blocks -locations pour vérifier la réplication.

⚡

2. PySpark — Lecture HDFS + Nettoyage

SparkSession connectée au NameNode. Filtrage des outliers, validation du schéma.

spark.read.parquet("hdfs://namenode:9000/data/raw") crée un DataFrame distribué — chaque partition Spark correspond à un bloc HDFS. Le nettoyage filtre : trip_distance ∈ ]0.1, 100[, fare_amount ∈ ]2.5, 500[, passagers 1–6.

⚙️ Adaptive Query Execution (spark.sql.adaptive.enabled=true) optimise automatiquement les shuffles.

🧪

3. Feature Engineering

Création des variables : trip_duration_min, hour_of_day, is_rush_hour, speed_mph, fare_per_mile…

Variables calculées :

• trip_duration_min = (dropoff - pickup) / 60 → variable cible
• hour_of_day, day_of_week, is_weekend → temporel
• is_rush_hour → heures 7-9h et 17-19h
• speed_mph, fare_per_mile → ratios

Le DataFrame est partitionné par year/month avant sauvegarde dans HDFS.

🤖

4. MLlib — Entraînement RandomForest

Pipeline distribué : VectorAssembler → RandomForestRegressor. Métriques : RMSE, MAE, R².

Le modèle est entraîné sur le cluster Spark (parallélisme des arbres). Le Pipeline MLlib enchaîne :

① VectorAssembler → combine les features en un vecteur dense
② RandomForestRegressor(numTrees=100) → 100 arbres parallèles
③ RegressionEvaluator → calcule RMSE, R²

⚙️ Le modèle est sauvegardé dans hdfs://namenode:9000/models/rf_model_v1 au format MLlib natif.

💾

5. Sauvegarde HDFS + CI/CD

Prédictions en Parquet partitionné. Pipeline CI : lint → tests → Docker build → push.

Les prédictions sont sauvegardées dans hdfs:///data/processed/predictions/year=2023/month=01/. Le partitionnement par date accélère les requêtes futures (partition pruning).

⚙️ CI/CD : GitHub Actions déclenche le pipeline sur chaque push. Le job test utilise file:// (pas de HDFS réel en CI), suivi du build Docker et du push sur Docker Hub.

🚀

6. Déploiement — SSH ou Azure

Option A : SSH sur VM Linux (Docker + Hadoop). Option B : Azure HDInsight + ADLS Gen2.

Option SSH : appleboy/ssh-action se connecte au serveur, fait un git pull, restart le cluster Docker, puis soumet le job Spark via spark-submit.

Option Azure : Upload du code sur ADLS2, soumission via API Livy ou SSH sur HDInsight en --deploy-mode cluster (le driver tourne sur le cluster, pas sur le runner GitHub).

💡 Rappel architectural : HDFS stocke les données, Spark les traite, MLlib entraîne les modèles, Git versionne le code, Docker reproductibilise l'environnement, GitHub Actions automatise, Azure déploie en production.

Architecture distribuée, blocs, réplication, NameNode/DataNode

├── data/
│ ├── raw/ # Données brutes Parquet (READ-ONLY)
│ │ ├── yellow_tripdata_2023-01.parquet [~500MB → 4 blocs × 128MB]
│ │ ├── yellow_tripdata_2023-02.parquet

🧠

NameNode

Cerveau du cluster. Gère les métadonnées : où sont stockés les blocs, quel DataNode possède quoi. Ne stocke PAS les données réelles. Si le NameNode tombe → cluster inaccessible (SPOF).

metadata only
💾

DataNode

Stocke les blocs de données réels (128 MB chacun). Envoie des heartbeats au NameNode toutes les 3 secondes. Si un DataNode tombe, HDFS re-réplique automatiquement les blocs manquants.

data storage
📋

Secondary NameNode

Fait des checkpoints périodiques du journal des transactions (EditLog). N'est PAS un NameNode de secours (contrairement à ce que le nom suggère !). Fusionne fsimage + edits.

checkpoint only
🔢

Facteur de réplication

Par défaut 3 copies de chaque bloc sur 3 DataNodes différents. Avec le rack awareness, 2 copies sur le même rack, 1 sur un rack distant. Tolérance aux pannes réseau.

durability × 3
📦

Taille des blocs : 128 MB

Bien plus grand que NTFS (4 KB) car HDFS est optimisé pour les lectures séquentielles de gros fichiers. Réduit le nombre de métadonnées à gérer par le NameNode.

vs NTFS 4KB

⚠️ Différence put vs copyFromLocal : hdfs dfs -put accepte aussi des sources HDFS, tandis que -copyFromLocal est strictement depuis le filesystem local. En pratique, ils sont équivalents pour les fichiers locaux.

Spark + MLlib — Traitement distribué

SparkSession, DataFrames distribués, pipeline MLlib RandomForest

🔗

Pourquoi hdfs:// et non file:// ?

En cluster multi-nœuds, chaque executor Spark tourne sur un serveur différent. file:// pointe vers le disque LOCAL de la machine — les autres nœuds ne peuvent pas y accéder. hdfs:// est le seul stockage partagé par tous.

🗂️

Partitions Spark ≈ Blocs HDFS

Quand Spark lit un fichier HDFS, il crée 1 partition par bloc (128 MB). Un fichier de 500 MB = ~4 partitions = 4 tâches parallèles. Après repartition(8), on force 8 partitions pour mieux paralléliser.

🧠

df.cache() — Pourquoi ?

Sans cache, chaque action Spark relit les données depuis HDFS. Avec df.cache(), le DataFrame est maintenu en mémoire RAM des executors. Les runs suivants dans le même job sont 10-100× plus rapides.

🔍

Predicate Pushdown

Parquet supporte le predicate pushdown : quand Spark filtre sur une colonne, il ne lit que les blocs Parquet concernés (row groups). df.explain(True) montre PushedFilters dans le plan physique.

PIPELINE MLLIB — RANDOMFOREST

from pyspark.ml import Pipeline
from pyspark.ml.feature import VectorAssembler
from pyspark.ml.regression import RandomForestRegressor
from pyspark.ml.evaluation import RegressionEvaluator

features = ['trip_distance', 'hour_of_day', 'day_of_week',
            'is_weekend', 'is_rush_hour', 'passenger_count']

assembler = VectorAssembler(inputCols=features, outputCol='features')
rf = RandomForestRegressor(
    featuresCol='features',
    labelCol='trip_duration_min',
    numTrees=100,
    maxDepth=10
)
pipeline = Pipeline(stages=[assembler, rf])

train, test = df.randomSplit([0.8, 0.2], seed=42)
model = pipeline.fit(train)       # entraînement distribué
preds = model.transform(test)    # prédictions

# Métriques
evaluator = RegressionEvaluator(labelCol='trip_duration_min')
rmse = evaluator.evaluate(preds, {evaluator.metricName: 'rmse'})
r2   = evaluator.evaluate(preds, {evaluator.metricName: 'r2'})

# Sauvegarde dans HDFS
model.write().overwrite().save('hdfs://namenode:9000/models/rf_v1')

# Rechargement
from pyspark.ml import PipelineModel
loaded = PipelineModel.load('hdfs://namenode:9000/models/rf_v1')
ℹ️ Feature Importances : model.stages[-1].featureImportances retourne un vecteur de scores. trip_distance est typiquement la feature la plus importante (~50%), suivi de hour_of_day (~20%) et is_rush_hour (~15%).

Le mot-clé needs

Définit les dépendances entre jobs. test: needs: lint signifie que le job test ne démarre que si lint réussit. Si lint échoue → test, build et push sont tous annulés. Pipeline "fail-fast".

dependency chain
💾

Cache pip (actions/cache)

Les dépendances Python sont mises en cache entre les runs. Un "run froid" (sans cache) télécharge tout : ~2-3 min. Un "run chaud" (cache valide) : ~15 secondes. Économie 80-90% du temps.

⚡ 10x faster
🐳

Condition push ≠ PR

push: github.event_name != 'pull_request' évite de pousser des images Docker non validées sur Docker Hub depuis une PR. On ne publie que depuis les branches protégées (main, develop) après review.

security
🧪

Tests sans HDFS réel

HDFS_NAMENODE='file://' redirige toutes les lectures/écritures HDFS vers le filesystem local du runner GitHub. Spark fonctionne en mode local[2]. Rapide, sans infrastructure externe.

file:// mock

WORKFLOW CI — STRUCTURE CLÉS

name: CI — Big Data Pipeline HDFS
on:
  push:
    branches: [main, develop, 'feature/**']
  pull_request:
    branches: [main, develop]

jobs:
  lint:                        # job 1
    runs-on: ubuntu-latest
    steps:
      - run: black --check src/ tests/
      - run: flake8 src/ --max-line-length=100

  test:
    needs: lint                # attend lint ✓
    env:
      HDFS_NAMENODE: 'file://'  # pas de vrai HDFS
    steps:
      - run: pytest tests/ --cov=src --cov-fail-under=80

  build:
    needs: test                # attend tests ✓
    steps:
      - uses: docker/build-push-action@v5
        with:
          push: ${{ github.event_name != 'pull_request' }}
💡 Git Flow dans ce projet : feature/* → PR → develop (CI obligatoire) → PR → main (CI + review) → CD déclenché automatiquement.

🔑

Clé SSH dédiée au déploiement

On génère une clé ED25519 séparée (jamais la clé personnelle ~/.ssh/id_ed25519). Si la clé de déploiement est compromise, on la révoque sans impacter l'accès personnel. La clé privée va dans les GitHub Secrets.

security isolation
☁️

WASB vs ADLS Gen2

WASB (Azure Blob) : stockage objet basique, sans hiérarchie réelle de dossiers. ADLS Gen2 (abfss://) : supporte les vrais répertoires, ACLs POSIX, et est optimisé pour Spark (meilleure latence, atomic rename).

prefer abfss://
⚙️

--deploy-mode cluster

En mode client, le driver Spark tourne sur la machine qui lance la commande (runner GitHub → timeout). En mode cluster, le driver tourne sur le cluster HDInsight → le runner peut se déconnecter, le job continue.

resilience
💰

Optimisation coûts Azure

HDInsight est facturé à l'heure. Entre deux déploiements, on peut az hdinsight resize --workernode-count 0 pour éteindre les workers. Le cluster reste provisionné mais ne coûte que le stockage ADLS2.

cost optimization

ROLLBACK AUTOMATIQUE — SSH

# Dans le script SSH du workflow CD :
PREV_IMAGE=$(docker inspect --format='{{.Config.Image}}' jupyter-spark 2>/dev/null || echo "")
docker-compose up -d
sleep 30

# Health check
if ! curl -f http://localhost:8080 2>/dev/null; then
  echo "❌ Health check failed — rollback..."
  if [ -n "$PREV_IMAGE" ]; then
    sed -i "s|image:.*|image: $PREV_IMAGE|" docker-compose.yml
    docker-compose up -d
    echo "✅ Rolled back to $PREV_IMAGE"
  fi
  exit 1
fi

Glossaire technique

Tous les termes clés du TP, expliqués simplement

🔵 HDFS — Hadoop Distributed File System
Système de fichiers distribué qui découpe les gros fichiers en blocs de 128 MB, répliqués sur plusieurs machines. Conçu pour stocker des pétaoctets avec tolérance aux pannes.
⚡ Spark / PySpark
Moteur de traitement distribué en mémoire. Lit depuis HDFS, distribue le calcul sur plusieurs machines (executors), 100x plus rapide que MapReduce pour les données en RAM.
🤖 MLlib
Bibliothèque de Machine Learning distribuée de Spark. Les algorithmes (RandomForest, etc.) s'exécutent en parallèle sur le cluster. Les modèles sont sauvegardés dans HDFS.
🐳 Docker / docker-compose
Containerisation : chaque service (NameNode, DataNode, Spark Master, Jupyter) tourne dans un container isolé. docker-compose orchestre l'ensemble du cluster localement.
🤖 CI/CD — Continuous Integration / Delivery
CI : automatise les tests à chaque push. CD : automatise le déploiement sur chaque merge dans main. GitHub Actions est le moteur qui exécute ces workflows.
☁️ Azure HDInsight
Cluster Hadoop/Spark managé par Microsoft Azure. Provisionné en ~15 min, facturé à l'heure, avec ADLS Gen2 comme stockage distribué (équivalent cloud de HDFS).
📦 Parquet
Format de fichier columnar binaire. Compression native, predicate pushdown, lecture partielle des colonnes. Bien plus efficace que CSV pour les workloads analytiques.
🔀 Shuffle
Redistribution des données entre les partitions Spark (lors d'un groupBy, join, etc.). Opération coûteuse car implique des transferts réseau. AQE optimise automatiquement les shuffles.
🔒 GitHub Secrets
Variables chiffrées stockées dans GitHub, injectées dans les workflows GitHub Actions. Jamais visibles dans les logs. Contiennent : clés SSH, tokens Docker, credentials Azure.
🌿 Git Flow
Convention de branches : main (production), develop (intégration), feature/* (nouvelles fonctionnalités). Les PRs vers main déclenchent le CI ; les merges déclenchent le CD.
📊 AQE — Adaptive Query Execution
Optimisation Spark (≥3.0) qui adapte le plan d'exécution en temps réel selon les statistiques collectées pendant l'exécution. Réduit le nombre de partitions de shuffle, corrige les skews.
🧪 pytest + coverage
Framework de tests Python. --cov=src mesure quelle proportion du code est exécutée par les tests. Le seuil de 80% est imposé par le CI (--cov-fail-under=80).
